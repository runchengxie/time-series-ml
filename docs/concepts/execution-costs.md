# 成本、执行假设与边界

本页解决什么：把项目当前的交易成本、执行假设和适用边界放到一页里。
范围：回测层的 TCA 假设；实际执行由 `quant-execution-engine` 负责。
适合谁：正在看回测结果、想知道哪些指标是执行近似、哪些假设需要在实际交易中重新验证的读者。
读完你会得到什么：当前成本模型、已知限制和与执行引擎的分工。
相关页面：`docs/methodology.md`、`docs/backtest.py`

## 先说结论

walk-forward 回测使用简化的交易成本模型。它已经比裸收益更有参考价值，但仍不是机构级 TCA。主要局限：

- 假设当日收盘价成交
- 单向成本 = `backtest_cost_bps / 2`（往返成本的一半）
- 未模拟滑点、订单簿、涨跌停无法成交
- 固定仓位（等权 100 股/笔）

如果你的策略换手率很高或者标的流动性较差，回测 Sharpe 会高估真实表现。

## 当前成本模型

### 买卖分边成本

`--backtest-buy-cost-bps` 和 `--backtest-sell-cost-bps` 分别控制买入和卖出的交易成本。

三种设置方式（优先级从高到低）：

1. 显式设置 `--backtest-buy-cost-bps` 和 `--backtest-sell-cost-bps`
2. 仅设置 `--backtest-sell-cost-bps`（买入=0，适用于 A 股印花税仅在卖出时收取）
3. 仅设置 `--backtest-cost-bps`（遗留，买卖各扣一半）

对于 A 股，推荐：`--backtest-sell-cost-bps 5`（印花税 5bps 卖出单向），`--backtest-buy-cost-bps` 可加佣金（如 1-2 bps）。

### 成本组成假设

默认 5 bps 往返成本的粗略分解（仅供参考，不是实际券商费率）：

| 成本项 | 比例 | 说明 |
| --- | --- | --- |
| 佣金 | ~1 bps | A 股券商佣金普遍在万 1-万 2.5 |
| 印花税 | 5 bps（卖出单向） | 约等于往返 2.5 bps |
| 滑点 | ~1.5 bps | 剩余部分粗略覆盖 |

如果你需要更准确的成本估算，调整 `--backtest-cost-bps`。

## 执行假设

| 假设 | 说明 | 局限 |
| --- | --- | --- |
| 成交价格 | 当日收盘价 | 实际可能以开盘价、盘中价或无法以收盘价成交 |
| 交易方向 | 只做多（无做空） | 熊市只能空仓等待，无对冲手段 |
| 仓位管理 | 等权 100 股/笔 | 无资金曲线动态调整、无仓位集中度控制 |
| 涨跌停处理 | `--no-price-limit-filter` 禁用，默认启用 | 涨停无法买入、跌停无法卖出，信号自动跳过 |
| 涨跌停识别 | 按板块自动判定（主板±10%，创业板/科创板±20%，北交所±30%） | 使用 pre_close 和 close 判定 |
| 流动性过滤 | `--min-daily-amount`（默认 0=不启用） | 使用日成交额 `amount` 列，低于阈值标记为不可交易 |
| 复权价格 | 未使用 | 回测收益计算未做复权调整 |
| 幸存者偏差 | 数据只含现存股票 | 已退市标的被排除，回测结果可能高估 |

## 与 quant-execution-engine 的分工

| 层面 | time-series-ml | quant-execution-engine |
| --- | --- | --- |
| 角色 | 研究侧策略评估 | 执行侧成交模拟和风控 |
| 成本模型 | 固定 bps | 分佣金+印花税+滑点+冲击成本 |
| 成交模拟 | 日线收盘价 | 可模拟订单簿和涨跌停约束 |
| 仓位管理 | 等权 100 股 | 资金曲线动态调整、风控限制 |
| 输出 | 回测统计指标 | 真实成交记录和审计日志 |

研究阶段用 time-series-ml 的回测判断策略方向是否有 alpha。确认方向后，交给 `quant-execution-engine` 做更细粒度的成交模拟和仓位管理。

## 当前实现的适用边界

以下场景里，当前做法通常已经够用：

- 日频方向预测的策略初筛
- 主要关心信号 IC 和排序能力是否为正
- 换手率 < 100/年、标的中等以上流动性

以下场景里，它就不够了：

- 换手率 > 200/年 → 交易成本假设不成立
- 标的小盘低流动性 → 冲击成本远高于模型假设
- 需要评估资金容量上限
- 需要区分开盘成交和收盘成交的差异
- 需要真实分红再投资和税收处理

## 推荐的下一步

1. 如果回测 Sharpe 接近或低于 0.3，先检查换手率是否过高
2. 如果换手率合理但 Sharpe 仍低，检查 `--regime` 分拆，看是否只在牛市有信号
3. 如果信号看起来不错，用 `--backtest-cost-bps 10` 或 `15` 做压力测试
4. 确认策略方向后，交给 `quant-execution-engine` 做执行层验证
